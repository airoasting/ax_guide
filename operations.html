6개월 운영 마일스톤

월별 테마 · 핵심 활동 · 끝낼 산출물

TF 임기 6개월을 월별 테마로 나눕니다. 각 달에는 한 가지 큰 테마만 둡니다. 두 가지 이상을 한 달에 묶으면 둘 다 어중간하게 끝납니다. 좌측은 그 달의 테마, 우측은 핵심 활동과 끝낼 산출물입니다. 산출물은 다음 달의 입력이 되도록 짭니다.

LLM 선택 기준 3가지

보안 적합성 · 도메인 성능 · 운영 비용

표준 LLM 선정은 출범 1개월 안에 끝냅니다. 멤버마다 다른 모델을 쓰면 보안 검토를 매번 다시 받아야 하고, 산출물의 품질 편차도 커집니다. 선택 기준은 세 가지로 충분합니다. 보안 적합성이 통과 조건이고, 도메인 성능과 운영 비용이 가중치입니다. 세 가지를 한 번에 점수화해 표준 1개와 보조 1개를 정합니다.

기준 1
보안 적합성으로
1차 거릅니다
영업기밀과 개인정보를 다루는 회사라면 학습 비활용 옵션, 데이터 처리 위치(리전), 사내 SSO 연동, 감사 로그가 1차 통과 조건입니다. 한 항목이라도 막히면 후보에서 빼고 다음으로 넘어갑니다. 사내 정보보안·법무가 처음부터 결정 회의에 참여해야 같은 검토를 두 번 받지 않습니다.
점검 항목학습 비활용, 데이터 리전, SSO/SCIM, 감사 로그, DLP 연동, 모델 카드 공개 여부.
기준 2
도메인 성능을
실제 업무로 평가합니다
벤치마크 점수보다 자기 회사의 실제 업무 샘플로 평가합니다. 빅 베팅 후보 영역에서 대표 과제 5~10건을 PoC 데이터셋으로 묶어 두 모델 이상에 같은 프롬프트로 돌립니다. 한국어 품질, 도구 호출(Function Call), 컨텍스트 윈도우, 추론 안정성을 함께 봅니다. 회의실 시연이 아니라 데이터 기반 비교 문서가 결과입니다.
비교 차원한국어 자연스러움, 도구 호출 안정성, 긴 문맥(20만 토큰 이상) 처리, 코딩 능력, 추론 일관성.
기준 3
운영 비용을
월간 예측치로 봅니다
토큰 단가만 비교하지 않습니다. 빅 베팅이 본격 가동되면 월 사용량이 PoC의 10~50배가 됩니다. 월간 예측 비용, 캐싱 적용 여부, 배치 할인, 멤버십(엔터프라이즈)·종량제 구조를 함께 산정합니다. 표준 모델은 본 작업 대다수를, 보조 모델은 비용 절감이나 특수 과제에 한정해 쓰는 이원 구조가 일반적입니다.
비용 변수토큰 단가, 캐시 할인, 배치 할인, 엔터프라이즈 정액제, 한도 초과 정책, 환율 변동.

주요 LLM 비교와 권고 구성

상용 모델 4종 · 강점 · 주의점 · 적합한 자리

아래 표는 2026년 기준 일반 기업 TF가 가장 자주 후보로 올리는 상용 모델 네 종을 정리한 것입니다. 모델 성능은 분기마다 빠르게 바뀌므로 절대 순위로 받지 않습니다. 자기 회사 PoC 결과와 함께 가중치를 매기는 비교 시작점으로 씁니다. 권고 구성은 표준 모델 1개와 보조 모델 1개를 두는 단순한 이원 구조이며, 멀티 모델 라우팅은 2단계 이후 도입을 권장합니다.

모델 강점 주의점 적합한 자리
ClaudeAnthropic 긴 문맥 추론과 코드·문서 작성 안정성이 강점입니다. 도구 호출과 에이전트 워크플로우에서 결과 일관성이 높습니다. Constitutional AI 기반의 안전성 정책이 분명합니다. 한국 리전이 아직 제한적입니다. 일본·미국 리전 사용 시 사내 데이터 정책 검토가 필요합니다. 가격대가 GPT 계열보다 일반적으로 높습니다. 긴 보고서·계약서·코드 작성과 에이전트 워크플로우의 표준 모델로 적합합니다. AX 1단계 표준 1순위 후보.
GPTOpenAI 생태계가 가장 넓고 사내 도구 연동(Microsoft 365·Copilot) 사례가 가장 많습니다. 멀티모달(이미지·음성)과 실시간 API가 강합니다. Azure OpenAI를 통한 엔터프라이즈 계약이 성숙합니다. 모델 버전 교체 주기가 빠르고 출력 톤이 자주 바뀝니다. 동일 프롬프트의 일관성을 회사 기준선으로 잡으려면 버전을 명시적으로 고정해야 합니다. Microsoft 365가 사내 표준인 회사의 표준 모델로 적합합니다. 멀티모달이 필요한 자리에 잘 맞습니다.
GeminiGoogle Google Workspace 연동과 초장문맥(200만 토큰급)이 강점입니다. 검색·지도 등 구글 자체 데이터 연결이 손쉽습니다. Vertex AI를 통한 데이터 거버넌스 옵션이 풍부합니다. 한국어 톤이 모델 버전별로 편차가 큽니다. 도구 호출과 함수 응답의 일관성이 GPT·Claude보다 낮다는 평이 자주 보입니다. 엔터프라이즈 도입 사례 폭이 좁습니다. Google Workspace가 사내 표준이거나 초장문맥이 핵심인 자리에 적합합니다. 보조 모델로 두는 회사가 많습니다.
국산·sLLMHyperCLOVA X 등 국내 데이터 리전과 한국어 문화 맥락의 처리가 강점입니다. 정부·공공·금융 규제 환경에서 사내 IT 검토 통과가 빠릅니다. 도메인 특화 파인튜닝의 사례가 늘고 있습니다. 고난도 추론과 영어 코드 작성에서 글로벌 상위 모델과 격차가 있습니다. 도구 호출 API 성숙도가 아직 후발입니다. 자체 호스팅 옵션은 운영 부담이 큽니다. 국내 개인정보·금융 규제 적용 업무의 보조 모델로 적합합니다. 단독 표준보다는 글로벌 모델과 이원 구성을 권장합니다.
권고 1

1단계는 표준 1개 + 보조 1개의 이원 구조

표준 모델 하나를 두면 회의 어휘와 산출물 톤이 통일됩니다. 보조 모델은 표준이 약한 영역(비용 절감, 특정 모달리티, 규제 업무)에만 한정해 씁니다. 처음부터 3개 이상을 동시에 굴리면 운영 비용이 학습 비용보다 빠르게 증가합니다.

권고 2

멀티 모델 라우팅은 2단계 이후로 미룹니다

라우터 게이트웨이(예: LiteLLM, OpenRouter, 사내 자체 게이트웨이)로 모델을 동적으로 분기하는 구조는 매력적입니다. 다만 1단계에서는 라우팅 정책을 검증할 데이터가 부족합니다. 2단계 우선순위화 이후 빅 베팅별 요구가 분명해진 시점에 도입을 검토합니다.

권고 3

모델 사용 가이드를 한 페이지로 못박습니다

표준 모델 선정 후 즉시 사내 가이드를 1.0으로 작성합니다. 어떤 데이터까지 입력 가능한지, 어떤 산출물에 사용할 수 있는지, 외부 공유 시 어떤 표기를 붙이는지 세 가지가 핵심입니다. 가이드가 없는 모델은 도입한 적이 없는 모델과 같습니다.

운영 리듬과 KPI

회의체 4종 · KPI 6개

운영 계획의 마지막 한 장은 회의 캘린더와 KPI 한 묶음입니다. 회의는 네 가지 주기로 충분하고, KPI는 여섯 개를 넘기지 않습니다. 회의가 늘기 시작하면 그 시점에 회의 자체를 안건으로 올리고, KPI가 늘기 시작하면 가장 약한 두 개를 골라 일단 폐기합니다.

회의 1

코어 데일리 (15분)

매일 오전 같은 시간에 15분으로 끊습니다. 오늘 끝낼 한 가지, 막혀 있는 한 가지, 도움 요청 한 가지만 돌아가며 말합니다. 안건이 있으면 별도 회의로 빼고 데일리 시간을 늘리지 않습니다.

매일 15분 코어 전원
회의 2

주간 회고 (60분)

금요일 오후에 한 주를 닫습니다. KPI 6개의 변화를 숫자로 보고, 다음 주 우선순위 3개를 합의합니다. 회고의 마지막 10분은 항상 "이번 주에 폐기할 일"을 정하는 데 씁니다.

주 1회 60분 코어 + 멘토
회의 3

월간 스폰서 보고 (45분)

월 1회 스폰서에 직접 보고합니다. 보고 형식은 항상 같은 한 장입니다. 이번 달 끝낸 산출물, 다음 달 약속, 보류·가속 결정이 필요한 항목. 45분 안에 끝나지 않으면 운영 계획이 흐려진 신호입니다.

월 1회 45분 스폰서 + 리드
회의 4

분기 외부 사례 학습 (반나절)

분기 1회 반나절을 외부 사례 학습에 씁니다. 컨설팅펌 리포트 1편, 동종 산업 사례 1편, 글로벌 빅테크 발표 1편을 코어가 돌아가며 발제합니다. 같은 자리에 스폰서가 동석해야 학습이 다음 분기 계획에 반영됩니다.

분기 1회 반나절 코어 + 스폰서

KPI 6개 (월간 갱신)

한 페이지에 들어가는 6개로 한정합니다. 셋은 활동량, 셋은 결과입니다. 7개부터는 추적이 무너집니다.

활동 KPI 3개
결과 KPI 3개
Quick Win 신규 가동 건수
월별로 가동된 신규 Quick Win 수. 5건 미만이면 부트캠프와의 짝이 풀린 신호입니다.
Quick Win 30일 동료 사용 흔적
가동된 Quick Win 중 동료 5명 이상이 직접 써본 비율. 60% 미만이면 다음 달 산출물 단위를 줄입니다.
Skill 문서 등록 수
사내 위키에 신규 등록된 Skill 문서 누적 수. 자산이 회사 안에 남는 속도입니다.
채택된 빅 베팅 후보 수
코어가 만든 자료 중 다음 단계 후보로 채택된 건수. 임기 종료 시 3~5건이 목표입니다.
월간 LLM 사용량과 비용
표준·보조 모델 합산 토큰과 비용. 한도의 80% 도달 시 다음 달 사용 패턴을 재설계합니다.
챔피언 후보 식별 수
임기 종료 시 자기 부서로 복귀시킬 챔피언 후보 누적 수. 6개월 임기 기준 5~9명이 정상 범위입니다.

흔한 실패 패턴과 대응

운영 계획을 무너뜨리는 5가지 함정

아래는 운영 계획을 짜고도 첫 90일 안에 자주 보이는 실패 패턴입니다. 좌측은 흔히 나타나는 모습, 우측은 같은 상황을 다르게 푸는 방법입니다. 3개 이상이 동시에 보이면 운영 계획 자체를 다시 점검할 신호입니다.

자주 나타나는 함정
대응 방향
"운영 계획이 PPT 50장이 됩니다." 형식이 두꺼워지면 정작 코어가 한 번도 펴보지 않습니다. 보고용 문서로 굳어 운영에서 멀어집니다.
운영 계획은 한 페이지로 묶습니다. 마일스톤·LLM·회의체·KPI 네 블록이 한 화면에 들어가야 매일 참고됩니다. 부속 자료는 별첨으로 둡니다.
"LLM 선정이 한 분기를 잡아먹습니다." 모델 비교에 두 달, 보안 검토에 한 달이 흐릅니다. 그 사이 멤버 각자가 익숙한 도구를 임시로 쓰며 검토 부담만 누적됩니다.
선정 기한을 1개월로 못박습니다. 보안·법무·IT를 같은 회의에 부르고, PoC 비교는 사내 업무 샘플 5~10건으로 한정합니다. 완벽한 비교가 아니라 첫 선택을 빠르게 합니다.
"마일스톤이 임기 종료까지 그대로 갑니다." 3개월 차에 외부 환경이 바뀌었는데도 1개월차에 적은 계획을 그대로 들고 갑니다.
분기마다 마일스톤을 다시 봅니다. 외부 환경(모델 출시·규제·경쟁사)과 내부 학습(부트캠프·Quick Win)에 따라 남은 3개월을 조정합니다. 다만 KPI는 함부로 갈지 않습니다.
"회의가 늘어납니다." 데일리·주간·월간·분기 외에 정기 외 회의가 생기고, 한 주에 회의가 8건을 넘기 시작합니다.
회의 추가는 주간 회고에서만 결정합니다. 새 회의를 더하려면 기존 회의 하나를 폐기해야 합니다. 회의 합계가 늘면 그 자체를 다음 회고 안건으로 올립니다.
"KPI가 14개로 늘어납니다." 보고할 때마다 측정 지표가 추가되고, 추적 자체가 풀타임 업무가 됩니다.
KPI 상한을 6개로 못박습니다. 새 지표를 더하려면 기존 지표 하나를 일단 폐기합니다. 추적 비용이 측정 가치를 넘기는 순간 KPI는 함정이 됩니다.
